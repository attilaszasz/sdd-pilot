# SDD Pilot тАФ Agent Context

This project uses **Spec-Driven Development**. Projects can optionally establish shared technical context before feature delivery begins.

## Project Bootstrap (optional)

```
Solution Architect тЖТ Init
```

- **Solution Architect** (`/sddp-systemdesign`) creates or refines `docs/sad.md` as the canonical Technical Context Document and registers it in `.github/sddp-config.md`
- **Init** (`/sddp-init`) establishes or amends `project-instructions.md` and preserves or adopts the registered Technical Context Document

Bootstrap is optional, but preferred when you want downstream phases to reuse shared architecture and stack context without manual copy/paste.

## Phase Order (strict)

```
Specify тЖТ Clarify тЖТ Plan тЖТ Checklist (optional) тЖТ Tasks тЖТ Analyze (optional) тЖТ Implement тЖТ QC
```

**Gates тАФ these are enforced, not suggested:**
- Never plan without `spec.md`
- Never generate tasks without `plan.md`
- Never implement without `tasks.md`
- Never mark a feature release-ready without `.qc-passed`
- `project-instructions.md` violations are always CRITICAL severity
- If checklists exist and any items are incomplete, implementation is blocked (override available)

## Conventions

### Feature folder naming
- New folders **must** use `00001-feature-name` format (5-digit prefix)
- Branch name `#####-feature-name` auto-resolves to `specs/<branch-name>/`

### Task format in `tasks.md`
```
- [ ] T### [P?] [US#?] {FR-###?} Description with file path
```
- `[P]` = parallelizable (safe to run in parallel with other `[P]` tasks)
- `[US#]` = maps to spec user story priorities P1, P2, P3
- `{FR-###}` = links task to functional requirement(s) from spec (e.g., `{FR-001}` or `{FR-001,FR-003}`)
- Phase order: optional Setup тЖТ optional Foundational тЖТ User Stories (by priority) тЖТ optional Polish
- Mark done: `- [ ]` тЖТ `- [X]`

### Priority system
- P1 = most critical, P1 alone should yield a viable MVP
- Each user story must be independently testable

### Project instructions (`project-instructions.md`)
- This is the **highest authority** in the SDD process
- Managed exclusively by `/sddp-init`

### QC artifacts
- `.completed` тАФ set by `/sddp-implement` when all tasks are done
- `.qc-passed` тАФ set by `/sddp-qc` when all quality checks pass
- `qc-report.md` тАФ detailed results from the QC run
- `manual-test.md` тАФ generated when manual verification is needed
- `checklists/.checklists` тАФ checklist queue generated by `/sddp-plan`; consumed by `/sddp-checklist`

### Project-level context artifacts
- `docs/sad.md` тАФ canonical Software Architecture Document / Technical Context Document when `/sddp-systemdesign` is used
- `.github/sddp-config.md` тАФ project-level registration for Product Document, Technical Context Document, checklist settings, and autopilot

### Implement + QC Loop (`/sddp-implement-qc-loop`)
- Optional convenience command тАФ combines `/sddp-implement` and `/sddp-qc` into a single continuous run
- Loops: implement тЖТ QC тЖТ (if QC fails) implement bug fixes тЖТ QC тЖТ тАж until QC passes
- Safety limit: **10 iterations** тАФ halts and reports if QC hasn't passed after 10 cycles
- Early halt triggers: user-chosen halt, `manual-test.md` generated, catastrophic implement failure, CRITICAL PI-only violations
- Does **not** change the phase order or gating rules тАФ it orchestrates the existing sub-skills

### Architecture pattern
- Workflow logic lives in `.github/skills/<name>/SKILL.md` (tool-agnostic)
- Tool-specific wrappers load these skills тАФ don't duplicate logic in wrappers

### Autopilot Mode

**Config**: `.github/sddp-config.md` тЖТ `## Autopilot` тЖТ `**Enabled**: true/false` (default: `false`)

When enabled, every feature-delivery SDD phase runs without user interaction тАФ all decision points use the recommended/default option. Every automatic decision is logged to `FEATURE_DIR/autopilot-log.md` for traceability.

**Command**: `/sddp-autopilot <feature description>`
- Orchestrates the full pipeline: Specify тЖТ Clarify тЖТ Plan тЖТ Checklist тЖТ Tasks тЖТ Analyze тЖТ Implement+QC
- Skill: `.github/skills/autopilot-pipeline/SKILL.md`
- Prompt: `.github/prompts/sddp-autopilot.prompt.md`
- Does **not** run project bootstrap phases like `/sddp-systemdesign` or `/sddp-init`

**Prerequisites** (enforced by Document Gate тАФ autopilot-only):
- **Product Document**: Registered in `sddp-config.md` under `## Product Document` тЖТ `**Path**:`. Must pass sufficiency check (тЙе3/5 content categories).
- **Technical Context Document**: Registered in `sddp-config.md` under `## Technical Context Document` тЖТ `**Path**:`. `docs/sad.md` created by `/sddp-systemdesign` is the preferred source. The document must pass sufficiency check (тЙе3/5 content categories).
- **Autopilot Enabled**: `**Enabled**: true` in config.

**Halt conditions** (pipeline stops immediately):
1. CRITICAL `project-instructions.md` violation at any phase
2. Implement-QC loop exhausted (10 iterations without QC passing)
3. `manual-test.md` generated (requires human verification)
4. Expected gate artifact missing after a phase
5. Feature already complete (`.qc-passed` exists)
6. Document sufficiency check failure

**Autopilot guards**: Each skill has conditional guards at user interaction points. When `AUTOPILOT = true`, the guard selects the default, logs the decision, and skips the prompt. Guard IDs are documented in each skill file (e.g., S1, P1, K1, I1, Q2, A1).

**Audit artifact**: `FEATURE_DIR/autopilot-log.md` тАФ records every automatic decision with timestamp, phase, decision point, chosen value, and rationale.

## Continuous Execution Policy

Execute all routine operations (file I/O, build/test/lint commands, git, task checkboxes, marker files, local package installs) **without pausing**. Only confirm: ambiguous requirements, system-level installs, destructive ops, or actions outside the project. Report at phase boundaries only.
